# 基于 RAGFlow 的多 Agent 食材推荐菜肴项目计划

**项目周期：** 7 天 × 6 小时/天（总计 42 小时）  
**文档版本：** v1.0  
**适用范围：** 课程项目、创新实践、原型验证  
**推荐实施形态：** RAGFlow 知识库 + 1 个总控 Agent + 3 个专业 Agent + 1 个确定性规则/评分模块 + 轻量前端

> **项目成功定义：** 用户用自然语言提供现有食材和约束后，系统在 12 秒左右返回 1～3 道有知识库依据、优先可直接制作的菜肴，清楚标注缺失食材、步骤、替代建议与安全提醒；50 条验收用例整体通过率不低于 85%。

## 1. 项目背景

家庭做饭的常见障碍不是“没有菜谱”，而是用户只知道手头有什么，却难以快速判断能做什么、还缺什么、是否符合时间和忌口。普通大模型可以生成菜谱，但容易忽略真实食材约束、虚构用量或遗漏安全事项。本项目以 RAGFlow 为知识检索与 Agent 编排平台，把结构化菜谱知识库、确定性匹配规则和多 Agent 协作结合起来，形成“输入理解—检索—匹配—生成—审查—展示”的闭环。

项目只有 42 个有效工时，因此采用可演示、可测试、可扩展的 MVP：先保证推荐有依据、排序合理、输出稳定，再考虑图片识别、营养分析和长期画像等扩展能力。

## 2. 项目目标与边界

### 2.1 核心目标

1. 接收自然语言食材输入，并识别人数、时间、难度、口味、忌口、厨具等约束。
2. 标准化食材名称，处理常见别名、重复项、模糊数量和基础调料。
3. 从 80～120 道结构统一的家常菜谱中检索候选项。
4. 依据核心食材覆盖率、检索相关度和用户约束进行确定性重排。
5. 返回 1～3 道菜，给出匹配理由、缺失项、步骤、时间、替代方案和安全提醒。
6. 通过食品安全 Agent 检查过敏原、忌口、生熟交叉污染和关键加热步骤。
7. 建立至少 50 条测试用例，并形成可重复演示和部署说明。

### 2.2 MVP 范围内

- 文本输入食材及偏好；
- 中文家常菜推荐；
- 食材同义词标准化；
- RAGFlow 检索与多 Agent 编排；
- 规则化匹配评分和缺失食材计算；
- 菜谱卡片、详细步骤、换一批；
- 过敏/忌口的硬约束检查；
- JSON 结构化输出和异常降级。

### 2.3 明确不做

- 食材图片识别、语音输入；
- 精确营养计算、医疗饮食诊断；
- 电商采购、支付、库存自动同步；
- 大规模自动爬取和版权不清的数据；
- 复杂长期用户画像、多人账号体系；
- 知识图谱、跨语言检索、复杂 Rerank；
- 生产级高并发、完整运维平台。

### 2.4 成功约束

- 若时间不足，优先保留：知识库质量、匹配评分、过敏约束、稳定 JSON、演示闭环。
- 前端和视觉效果可以降级，但不能以虚构菜谱或放松安全校验换取展示效果。
- 菜谱推荐必须能追溯到知识库记录；生成 Agent 仅负责改写与适配，不负责凭空创造核心菜谱。

## 3. 用户角色与典型场景

### 3.1 主要用户

- **家庭做饭者：** 希望快速利用现有食材完成一餐。
- **学生/租房用户：** 厨具有限、时间短、食材少。
- **有忌口或过敏需求者：** 需要排除特定食材，并看到风险提示。
- **项目演示者/维护者：** 需要查看检索、评分和审查结果以定位问题。

### 3.2 核心用户故事

1. 作为家庭做饭者，我输入“鸡蛋、番茄、土豆、青菜”，希望看到 20 分钟内能完成的 3 道菜。
2. 作为厨具有限的学生，我声明“只有炒锅”，希望系统不推荐必须使用烤箱的菜谱。
3. 作为花生过敏用户，我希望任何含花生或可能冲突的推荐都被拦截，而不是仅显示提醒。
4. 作为用户，我想展开某道菜查看按 2 人份组织的准备工作、步骤、火候和成熟判断。
5. 当没有完全匹配的菜谱时，我希望系统说明还缺哪些关键食材，并给出合理替代方案。
6. 当系统暂时无法检索时，我希望得到明确提示和可重试建议，而不是一段看似可信但无依据的菜谱。

## 5. 总体系统架构

```text
用户浏览器
  │ 食材、人数、时间、口味、忌口、厨具
  ▼
轻量前端（HTML/React/Vue，可先用 iframe）
  │ POST /api/recommend
  ▼
轻量后端 / 规则服务（FastAPI，可由 RAGFlow HTTP Request 调用）
  ├─ 输入校验与食材标准化
  ├─ 同义词/基础调料规则
  ├─ 覆盖率、缺失项与最终评分
  └─ JSON Schema 校验、日志与超时控制
  │ 标准化请求
  ▼
RAGFlow 工作流
  ├─ Categorize：意图分类
  ├─ ChefCoordinator：总控 Agent
  ├─ Retrieval：菜谱知识库检索
  ├─ RecipeMatcher：候选语义评估
  ├─ CookingGuide：做法整理与适配
  └─ FoodSafetyReviewer：安全/约束审查
  │
  ▼
结构化推荐 JSON → 前端菜谱卡片
```

### 5.1 技术选型

- **RAG 与编排：** RAGFlow；
- **模型：** 团队已可稳定调用的中文指令模型，优先保证结构化输出能力；
- **Embedding：** 与中文语料匹配的现有可用模型，项目期间不频繁更换；
- **规则服务：** Python + FastAPI；若已配置 RAGFlow Sandbox，可将部分逻辑放入 Code 组件；
- **前端：** 时间紧时使用 RAGFlow 嵌入；有前端成员时使用轻量 React/Vue 或原生 HTML；
- **数据：** Markdown 菜谱文件 + JSON 同义词表 + JSON Schema；
- **测试：** pytest/接口脚本 + 人工 UI 验收表。

### 5.2 关键设计原则

- LLM 负责理解、规划、语言组织；规则代码负责去重、计算、排序和格式校验。
- “过敏/忌口”是硬约束，“口味/难度”是软约束。
- 多 Agent 之间传 JSON，避免传递无边界的大段自然语言。
- 保留每阶段输入输出和耗时，便于在 7 天内快速定位问题。

## 6. 多 Agent 设计

### 6.1 ChefCoordinator（总控 Agent）

**职责：** 理解意图、检查必要信息、构造检索请求、调用工具/子 Agent、控制修订次数、整合最终输出。  
**输入：** `NormalizedRequest`。  
**输出：** `RecommendationResponse`。  
**关键规则：** 无食材时追问；最多返回 3 道；优先零核心缺失；不得宣称不存在的知识库来源；安全审查失败时仅允许一次修订，仍失败则移除该推荐。

### 6.2 RecipeMatcher（菜谱匹配 Agent）

**职责：** 基于候选菜谱语义判断核心/普通/可选食材，评估替代合理性和偏好符合度，为确定性评分提供特征。  
**输入：** 用户食材、约束、Retrieval 候选。  
**输出：** 候选特征列表，包括覆盖情况、冲突、替代建议、解释。  
**边界：** 不得只看菜名；不得自行把核心食材降级为可选；硬约束冲突必须标记为 `blocked=true`。

### 6.3 CookingGuide（烹饪指导 Agent）

**职责：** 将知识库步骤改写为清晰、可执行的流程，并按人数、替代项和厨具做有限适配。  
**输出结构：** 准备工作、编号步骤、火候、预计时长、成熟判断、易错点。  
**边界：** 不改变核心烹饪逻辑；不虚构精确克数；不添加用户没有且未标为缺失的关键食材。

### 6.4 FoodSafetyReviewer（安全审查 Agent）

**职责：** 检查过敏原与忌口冲突、肉蛋海鲜加热、生熟交叉污染、步骤遗漏和输入输出一致性。  
**输出：** `passed`、`severity`、`issues`、`revision_suggestions`。  
**策略：** 过敏/忌口冲突为阻断级；一般烹饪提醒为提示级；不提供疾病治疗或医学营养建议。

### 6.5 确定性规则模块

负责食材去重、同义词映射、基础调料识别、覆盖率、缺失项、评分排序、Schema 校验和候选去重。它不是 Agent，但构成系统稳定性的核心。

## 7. RAGFlow 工作流设计

### 7.1 主流程节点

```text
Begin
  → 输入校验/标准化（HTTP Request 或 Code）
  → Categorize（推荐 / 查做法 / 找替代 / 非目标问题）
  → ChefCoordinator
      → Retrieval（Top N 候选）
      → RecipeMatcher
      → 规则服务评分与排序
      → CookingGuide（Top 1～3）
      → FoodSafetyReviewer
      → 必要时修订一次
  → JSON Schema 校验
  → Message / API Response
```

### 7.2 分支逻辑

- **推荐菜肴：** 完整执行检索、匹配、评分、做法和安全审查。
- **查看做法：** 按菜名/`recipe_id` 精确检索，直接进入 CookingGuide 与安全审查。
- **寻找替代：** 读取原菜谱和缺失项，输出替代可行性及步骤变化。
- **非目标问题：** 简短说明系统能力，引导用户提供食材，不调用全部 Agent。

### 7.3 推荐检索参数初值

| 参数 | 初始值 | 调优方式 |
|---|---:|---|
| Similarity threshold | 0.20 | 召回不足降至 0.15；噪声过多升至 0.25 |
| Vector similarity weight | 0.40 | 菜名命中差则略降，语义召回差则略升 |
| Top N | 10 | 在 8～12 范围用测试集选择 |
| Rerank | 关闭 | MVP 稳定后再评估延迟收益 |
| Knowledge Graph | 关闭 | 不在 7 天范围内 |
| Cross-language | 关闭 | 中文单语项目无必要 |

### 7.4 变量传递约定

- 节点只接收所需字段，候选菜谱单条正文建议控制在 1,500 字以内。
- 所有 Agent 输出 JSON；禁止把思维过程作为正式字段。
- 每个节点记录 `request_id`、`stage`、`duration_ms`、`status` 和简化错误码。
- 总控重试上限为 1；全链路建议超时 15 秒，前端 12 秒显示“仍在处理”。

## 8. 知识库设计

### 8.1 数据规模与构成

MVP 准备 80～120 道常见家庭菜。建议分布：蛋类 10～15、豆制品 10～15、蔬菜 20～25、鸡肉 10～15、猪牛肉 15～20、鱼虾 8～12、主食/汤类 15～20；一道菜可多标签。

### 8.2 数据来源与治理

- 使用团队自编、公开授权或明确可用于项目的数据；记录来源和整理人。
- 不直接批量抓取来源不明的菜谱正文。
- 每道菜由一人录入、另一人抽查核心食材、步骤、过敏原和时间。
- 菜谱发生修改时更新 `version` 与 `updated_at`。

### 8.3 切分与索引

- 一道菜一个 Markdown 文件，尽量保持为单个完整 Chunk。
- 文件开头重复菜名、核心食材和标签，提升关键词召回。
- 同义词作为关键词或别名字段，如“西红柿/番茄”“马铃薯/土豆”。
- 不把所有菜谱合并为单一长文档。
- 上传后抽查 Chunk，确保步骤未与标题、食材表分离。

### 8.4 菜谱文件模板

```yaml
recipe_id: R001
title: 番茄炒蛋
cuisine: 家常菜
tags: [快手菜, 下饭菜, 鸡蛋类]
difficulty: 简单
estimated_time_min: 15
servings: 2
core_ingredients: [番茄, 鸡蛋]
seasonings: [食用油, 盐, 糖]
optional_ingredients: [葱]
equipment: [炒锅]
allergens: [鸡蛋]
version: 1.0
```

正文继续包含用量描述、准备、编号步骤、替代方案、成熟判断和注意事项。

## 10. API 与模块设计

### 10.1 推荐接口

**POST `/api/recommend`**  
请求：用户原始文本和表单约束。  
响应：`RecommendationResponse`。  
错误：`400 INVALID_INPUT`、`422 NO_INGREDIENT`、`503 RETRIEVAL_UNAVAILABLE`、`504 WORKFLOW_TIMEOUT`、`500 INVALID_MODEL_OUTPUT`。

### 10.2 其他接口

| 接口 | 用途 | MVP 优先级 |
|---|---|---:|
| POST `/api/normalize` | 标准化食材，便于独立测试 | P0 |
| POST `/api/score` | 计算候选评分并排序 | P0 |
| GET `/api/recipes/{id}` | 查看某道菜的完整知识库记录 | P1 |
| POST `/api/substitute` | 对指定缺失食材给出替代 | P1 |
| GET `/api/health` | 检查规则服务和 RAGFlow 连通性 | P0 |

### 10.3 后端模块

- `normalizer`：分词、去重、别名映射、类别和数量保留；
- `staples_policy`：基础调料白名单和用户开关；
- `scorer`：覆盖率、缺失项、硬约束过滤和排序；
- `ragflow_client`：调用工作流、设置超时、转换错误；
- `schema_validator`：验证并清理模型输出；
- `observability`：请求 ID、阶段耗时、错误码和脱敏日志。

## 11. 推荐评分逻辑

### 11.1 硬过滤

评分前先移除：明确过敏原冲突、明确忌口冲突、必需厨具不具备且无替代烹饪方式的候选。硬过滤结果不参与排名。

### 11.2 评分公式

```text
Base = 45 × 核心食材覆盖率
     + 20 × 归一化检索相关度
     + 15 × 用户偏好符合度
     + 10 × 时间符合度
     + 10 × 厨具符合度

Penalty = 25 × 核心食材缺失比例
        + 10（超过时间上限但不超过 20% 时）

FinalScore = clamp(round(Base - Penalty), 0, 100)
```

其中，各符合度取 0～1。若超过时间上限 20% 以上，候选直接降为低优先级；核心食材覆盖率低于 50% 时原则上不进入 Top 3，除非无其他候选且明确标注“需补购”。

### 11.3 排序与去重

按 `blocked`、`missing_core_count`、`FinalScore`、`estimated_time` 依次排序；菜名或核心食材高度相似的候选只保留一个。若零核心缺失候选不足 3 道，可补充最多 1 道仅缺 1 项核心食材的菜，并明确提示。

### 11.4 评分样例

| 菜谱 | 核心覆盖 | 缺失核心 | 检索相关度 | 约束情况 | 预期分 |
|---|---:|---|---:|---|---:|
| 番茄炒蛋 | 100% | 无 | 0.90 | 全符合 | 96 |
| 青菜鸡蛋汤 | 100% | 无 | 0.72 | 全符合 | 88 |
| 番茄土豆汤 | 100% | 无 | 0.66 | 全符合 | 85 |
| 番茄牛腩 | 50% | 牛腩 | 0.80 | 超时 | 35～45 |

## 12. 前端页面与交互

### 12.1 页面构成

1. **输入区：** 食材自然语言输入框；
2. **约束区：** 人数、15/30/60 分钟、难度、口味、忌口/过敏、厨具、基础调料开关；
3. **状态区：** 识别出的标准食材、加载阶段、可修改提示；
4. **结果区：** 最多 3 张菜谱卡片；
5. **详情区：** 展开准备、步骤、火候、成熟判断、安全提醒；
6. **操作区：** 换一批、查看详细做法、修改条件、复制菜谱。

### 12.2 菜谱卡片字段

菜名、匹配分、预计时间、难度、已用食材、缺失核心/可选食材、推荐理由、安全标识。缺失核心食材使用醒目标识；安全阻断的菜谱不展示。

### 12.3 状态设计

- 空状态：给出可直接点击的示例输入；
- 加载状态：显示“正在整理食材 / 检索菜谱 / 检查安全”等阶段；
- 无结果：建议放宽时间、补充食材或允许一种缺失项；
- 错误状态：保留用户输入，提供重试，不展示无依据的兜底菜谱；
- 移动端：卡片单列，步骤折叠，主要按钮保持可见。

## 13. 异常、隐私与安全策略

### 13.1 异常处理

| 异常 | 系统行为 | 用户提示 |
|---|---|---|
| 未输入食材 | 不调用检索 | 请至少提供一种现有食材 |
| 食材无法识别 | 保留原词并标记待确认 | 我识别到“X”，请确认它是否为食材 |
| 检索为空 | 放宽阈值重试 1 次 | 暂未找到完全匹配，可调整条件 |
| RAGFlow 超时 | 终止链路并记录阶段 | 服务繁忙，请稍后重试 |
| 模型 JSON 无效 | 自动修复/重试 1 次 | 若仍失败则返回结构化错误 |
| 安全审查失败 | 修订一次或移除候选 | 不展示存在硬冲突的菜谱 |
| 候选不足 3 道 | 返回实际数量 | 不为凑数虚构推荐 |

### 13.2 食品安全

- 过敏原、宗教/饮食忌口为硬约束；
- 生肉、生蛋、鱼虾需包含充分加热与避免交叉污染提醒；
- 不输出医疗诊断、疾病治疗和精确营养处方；
- 对孕妇、婴幼儿等高风险人群仅给通用提示，建议咨询专业人士；
- 安全 Agent 输出与最终结果一起留痕，便于复核。

### 13.3 隐私和日志

- MVP 不要求账号和长期画像；
- 日志仅保存请求 ID、标准化食材、阶段耗时和错误码；
- 忌口/过敏字段按敏感偏好处理，演示日志中脱敏；
- 不记录模型密钥、完整请求头或其他凭证。

## 14. 测试方案

### 14.1 测试层级

- **单元测试：** 同义词、去重、基础调料、缺失项、评分边界、Schema 校验；
- **知识库检索测试：** 20 组典型查询检查 Top N 召回；
- **Agent 契约测试：** 每个 Agent 的输入/输出 JSON、禁止项和错误处理；
- **端到端测试：** 从输入到卡片展示的 50 条用例；
- **人工安全复核：** 所有过敏、忌口、生肉、生蛋、海鲜场景；
- **性能测试：** 连续执行 10 次，记录 P50/P95 响应时间和失败率；
- **UI 测试：** 桌面与移动宽度下的空、加载、成功、无结果、错误状态。

### 14.2 50 条测试用例构成

| 类别 | 数量 | 示例 |
|---|---:|---|
| 标准食材组合 | 12 | 鸡蛋+番茄、土豆+青椒 |
| 别名/错别字/模糊量 | 8 | 西红柿、马铃薯、半个土豆 |
| 单一或大量食材 | 5 | 只有鸡蛋；一次输入 12 种食材 |
| 时间/难度/厨具 | 6 | 15 分钟、只用电饭锅 |
| 口味与忌口 | 5 | 不辣、素食、不吃香菜 |
| 过敏与安全 | 6 | 花生、鸡蛋、海鲜过敏 |
| 无匹配/异常 | 4 | 稀有食材、检索超时 |
| 追问与替代 | 4 | 换一批、替代牛奶、查看做法 |

### 14.3 关键断言

- Top 3 至少一道零核心缺失菜；
- 不出现用户排除的食材；
- 缺失核心项必须明确展示；
- JSON 字段完整、类型正确；
- 推荐步骤与知识库核心逻辑一致；
- 无检索依据时不得生成具体菜谱；
- 安全阻断场景通过率为 100%。

## 18. 风险与应对

| 风险 | 概率/影响 | 预警信号 | 应对措施 |
|---|---|---|---|
| RAGFlow/模型环境不稳定 | 中/高 | 第 1 天无法完成检索 | 当天解决；保留固定候选 JSON 用于前端联调 |
| 菜谱质量不一致 | 高/高 | Chunk 断裂、核心食材缺字段 | 模板校验、交叉抽查、先质量后数量 |

| Agent 输出 JSON 不稳定 | 中/高 | 解析错误超过 5% | Schema 约束、低温度、一次修复、规则兜底 |
| 排名受语义常见度误导 | 高/中 | 缺核心食材菜排在前面 | 强化确定性覆盖率和硬惩罚 |
| 过敏/忌口漏检 | 低/极高 | 专项用例失败 | 前后双重检查、硬过滤、失败不展示 |
| 数据版权不清 | 中/中 | 无法说明菜谱来源 | 使用自编/授权数据并登记来源 |
| 前端耗时失控 | 中/中 | 第 5 天仍未联通接口 | 立即降级 iframe 或原生 HTML |
| 范围蔓延 | 高/高 | 临时加入图片/营养/账号 | 按 P0/P1/P2 管理，第 7 天只修阻断问题 |
| 现场网络或服务故障 | 中/高 | 彩排偶发失败 | 备用本地部署、录屏、截图和固定结果 |

## 20. 演示流程

我有鸡蛋、番茄、半个土豆和一些青菜，两个人吃，
