# 后端两人分工

整个后端沿 LangGraph 数据流切成两条线，中间靠 `schemas.py` 做契约，互不干涉。

```
食材文本 ──→ ┌─── A线：规则管线 ───┐ ──→ ┌─── B线：Agent管线 ───┐ ──→ JSON响应
             │ normalize → retrieve │     │ match → guide → safety │
             │ → score → rank       │     │ → graph → api          │
             └──────────────────────┘     └────────────────────────┘
                   ↑ 接口契约 ↑              ↑ 接口契约 ↑
                   CandidateFeature          Recommendation
                   RecipeRecord              SafetyReport
```

## A线：规则管线

**一句话：** 输入食材文本，输出排序后的候选菜谱特征列表。纯 Python 逻辑，不调 LLM。

### 文件

| 文件 | 做什么 | 输入 → 输出 |
|---|---|---|
| `rules/normalizer.py` | 分词、别名映射、去重、分类 | `"西红柿、鸡蛋2个"` → `[RawIngredient(name="番茄",...)]` |
| `rules/staples.py` | 基础调料白名单（盐油酱醋葱姜蒜） | 判断某食材是否默认拥有 |
| `rules/scorer.py` | 候选特征提取 + 硬过滤 + 评分排序 | `(RecipeRecord, 用户食材, 约束)` → `CandidateFeature` |
| `retrieval/stub.py` | 从知识库检索候选菜谱 | `["鸡蛋","番茄"]` → `List[RecipeRecord]` |

### 不做什么

- 不调任何 LLM / API
- 不管 LangGraph、FastAPI
- 不生成做法步骤和安全提醒

### 对 B 暴露的接口

```python
# retrieval
retrieval.search(ingredient_names: list[str], top_n: int) -> list[RecipeRecord]

# scorer
build_feature(recipe, ingredients, allergens, excluded, flavor, time, equipment) -> CandidateFeature
score_and_rank(features: list[CandidateFeature]) -> list[CandidateFeature]
```

### 测试方式

```bash
python -X utf8 test_demo.py "鸡蛋、番茄"
# 看 Step 2~5：标准化 → 检索 → 匹配 → 评分
```

## B线：Agent管线

**一句话：** 拿 A 的候选特征，调 LLM 做语义匹配、做法生成、安全审查，最后通过 FastAPI 输出 JSON。

### 文件

| 文件 | 做什么 | 依赖 |
|---|---|---|
| `llm_client.py` | SiliconFlow 客户端封装（chat / chat_json / 多模态） | config |
| `agents/matcher.py` | LLM 语义匹配：覆盖判断、偏好评分、冲突检测 | llm_client + schemas |
| `agents/cooking_guide.py` | LLM 生成准备/步骤/火候/替代 | llm_client + schemas |
| `agents/safety.py` | 规则硬拦截 + LLM 增强审查 | llm_client + schemas |
| `graph.py` | LangGraph 8 节点编排，调用 A 和 B 的模块 | 全部 |
| `app.py` | FastAPI 入口：`/recommend` `/health` `/recipes/{id}` | graph |
| `routers/auth.py` | 注册/登录/JWT（MySQL） | database |
| `database.py` | MySQL 连接池 + users 表 | config |

### 不做什么

- 不关心食材怎么标准化、别名怎么映射
- 不关心菜谱怎么从知识库检索出来
- 不自己写评分排序逻辑

### 对 A 的依赖

```python
from rules.normalizer import normalize_ingredients   # 标准化
from rules.staples import get_staples                # 调料识别
from rules.scorer import build_feature, score_and_rank  # 特征+排序
from retrieval.stub import RetrievalStub             # 检索
```

## 接口契约（两人分界线）

A 提供，B 调用。改契约时先沟通。

### `CandidateFeature`（A 产出，B 消费）

```python
class CandidateFeature(BaseModel):
    recipe_id: str
    retrieval_score: float        # A: 检索相关度
    core_total: int               # A: 核心食材总数
    core_matched: int             # A: 匹配到的核心数
    missing_core: list[str]       # A: 缺失核心食材
    missing_optional: list[str]   # A: 缺失可选食材
    preference_score: float       # A: 偏好符合度 (0~1)，B 可用 LLM 覆盖
    time_fit: float               # A: 时间符合度 (0~1)，B 可用 LLM 覆盖
    difficulty_fit: float         # A: 难度符合度 (0~1)
    equipment_fit: float          # A: 厨具符合度 (0~1)
    blocked: bool                 # A: 硬过滤标记
    block_reasons: list[str]      # A: 阻塞原因
    final_score: int              # A: 最终评分 (0~100)
```

### `RecipeRecord`（A 产出，B 消费）

```python
class RecipeRecord(BaseModel):
    recipe_id: str
    title: str
    cuisine: str
    tags: list[str]
    difficulty: str
    estimated_time_min: int
    servings: int
    core_ingredients: list[str]
    seasonings: list[str]
    optional_ingredients: list[str]
    equipment: list[str]
    allergens: list[str]
    body: str                    # 做法正文
    retrieval_score: float
```

### `SafetyReport`（B 产出）

```python
class SafetyReport(BaseModel):
    passed: bool
    severity: str                # "none" | "warning" | "blocked"
    issues: list[str]
    revision_suggestions: list[str]
```

### `RecommendationResponse`（B 产出，前端消费）

```python
class RecommendationResponse(BaseModel):
    request_summary: RequestSummary
    recommendations: list[Recommendation]   # 1~3 道
    blocked_recipes: list[dict]
    follow_up_question: str | None
    trace_id: str
```

## 开发顺序

两条线可以并行开发，但联调时有先后依赖：

```
第1~2天：A 先跑通 normalizer + scorer + retrieval stub
          B 搭好 FastAPI 骨架 + LLM client + 三个 Agent 独立测试

第3天：  两边对一下 schemas.py（契约评审）
          A 给 B 提供稳定的 build_feature / score_and_rank 接口

第4天：  B 把 graph.py 串起来，调 A 的接口
          联调端到端

第5~7天：各自完善自己的部分，不用等对方
```

### 各自独立测试，互不阻塞

| A 独立可做的事 | B 独立可做的事 |
|---|---|
| 写 normalizer 别名表 | 搭 FastAPI + CORS |
| 调检索 recall | 写 3 个 Agent 的 prompt |
| 评分公式单元测试 | 接 SiliconFlow 测试 JSON 输出 |
| 菜谱数据录入清洗 | 写 graph.py 骨架（先用假数据） |
| 换 RAGFlow 真实检索 | 写 auth 注册登录 |
| | 写前端对接的 mock 数据 |

### 需要一起做的事

- 评审 `schemas.py`（1小时）
- 端到端联调（半天）
- 安全专项测试（半天）
